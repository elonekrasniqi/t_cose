19/* This is an "abstract base class" for all signers
20 * of all types for all algorithms. This is the interface
21 * and data structure that t_cose_sign_sign knows about to be able
22 * to invoke each signer regardles of its type or algorithm.
23 *
24 * Each concrete signer must implement this. Each signer
25 * also implements a few methods of its own beyond this
26 * that it needs to work like those for initialization and
27 * setting the key.
28 *
29 * The reason signers are abstracted out as they are here is in
30 * anticipation of more complicated signers that support things like
31 * counter signing, post-quantum signatures and certificate
32 * hierarchies. A signer may support only one signing algorithm, but
33 * that is not required. For examples the "main" signer supports
34 * ECDSA and RSA because they are very similar. The EdDSA signer is
35 * separate because it doesn't involve a hash. Counter signature are
36 * too complicated to support with custom parameters so they should
37 * implement a signer (need to validate the interface will work for
38 * them).
39 *
40 * t_cose_signer_callback is the type of a function that every
41 * signer must implement. It takes as input the context
42 * for the particular signer, the hash to sign and
43 * the encoder instance. The work it does is to produce
44 * the signature and output the COSE_Signature to the
45 * encoder instance.
46 *
47 * t_cose_signature_sign_h_callback is a callback that signers
48 * that support COSE_Sign1 must implement in addition to
49 * t_cose_signer_callback. It returns the
50 * headers that have to go in the COSE_Sign1 body.
51 *
52 * This design allows new signers for new algorithms to be added
53 * without modifying or even recompiling t_cose. It is a
54 * design that allows outputting a COSE_Sign that has multiple
55 * signings by multiple aglorithms, for example an ECDSA signature and
56 * an HSS/LMS signature.
57 *
58 * Because this design is based on dynamic linking it gives some help
59 * dealing with code size and dependency on crypto libraries. If you
60 * don't call the init function for a signer or verifier it won't be
61 * linked No disabling #define needed. You can just leave out the
62 * source files for signers/verifiers you don't want and all will
63 * compile and link nicely without having to managed a #define. (This
64 * doesn't work to elimiate RSA if you use only ECDSA though because
65 * they both go through the same layer in the crypto adaptation
66 * layer. It does work for EdDSA).
67 *
68 * What's really going on here is a bit of doing object orientation
69 * implementned in C. This is an abstract base class, an object that
70 * has no implementation of it's own. Each signer type, e.g., the
71 * main signer, inherits from this and provides an
72 * implementation. The structure defined here holds the vtable for the
73 * methods for the object. Only two methods happen to be needed. It's
74 * called a "callback" here, but it could also be called the abstract
75 * sign method.
76 *
77 * Since C doesn't support object orientation there's a few tricks to
78 * make this fly. The concrete instantiation (e.g., the main signer)
79 * must make struct t_cose_signature_sign is the first part of its
80 * context and there will be casts back and forth between this
81 * abstraction and the real instantion of the signer. The other trick
82 * is that struct here contains a pointer to a function and that makes
83 * up the vtable, something that C++ would do for you.
84 */
