25/*
26 * API Design Overview
27 *
28 * t_cose is made up of a collection of objects (in the
29 * object-oriented programming sense) that correspond to the main
30 * objects defined in CDDL by the COSE standard (RFC 9052). These
31 * objects come in pairs, one for the sending/signing/encrypting
32 * side and the other for the receiving/verifying/decrypting
33 * side. Following is a high-level description of all of these and how
34 * they connect up to each other.
35 *
36 * Some of this is implemented and some of this is a design proposal,
37 * so it is subject to some change, renaming and such as the
38 * implementation completes.
39 *
40 *
41 * COSE_Sign and COSE_Sign1
42 *
43 * t_cose_sign_sign() and t_cose_sign_verify() are the pair that
44 * implements both COSE_Sign and COSE_Sign1 COSE messages.
45 *
46 * They rely on implementations of t_cose_signature_sign and
47 * t_cose_signature_verify to create and to verify the
48 * COSE_Signature(s) that are in a COSE_Sign. They are also used to
49 * create the signature for COSE_Sign1. These two are an abstract
50 * base class they are just an interface without an implementation.
51 *
52 * t_cose_headers_decode() and t_cose_headers_encode() are also used
53 * by the t_cose_sign pair. These process both the protected and
54 * unprotected header parameter buckets called Headers in COSE.
55 *
56 *
57 * COSE_Encrypt and COSE_Encrypt0
58 *
59 * t_cose_encrypt_enc() and t_cose_encrypt_dec() are the pair for
60 * COSE_Encrypt and COSE_Encrypt0.
61 *
62 * t_cose_headers_decode() and t_cose_headers_encode() are used for
63 * the header parameters.
64 *
65 * This makes use of implementations of t_cose_recipient_enc and
66 * t_cose_recipient_dec for COSE_recipient used by COSE_Encrypt. They
67 * are not needed for COSE_Encrypt0.
68 *
69 *
70 * COSE_Mac and COSE_Mac0
71 *
72 * t_cose_mac_auth() and t_cose_mac_check() are the pair for COSE_Mac
73 * and COSE_Mac0.
74 *
75 * t_cose_headers_decode() and t_cose_headers_encode() are used for
76 * the header parameters.
77 *
78 * For COSE_Mac, t_cose_recipient_enc() and t_cose_recipient_dec()
79 * implement COSE_recipient. (I’m pretty sure sharing t_cose_recipient
80 * between COSE_MAC and COSE_Encrypt can work, but this needs to be
81 * checked by actually designing and implementing it). These are not
82 * needed for COSE_Mac0.
83 *
84 *
85 * COSE_Message
86 *
87 * t_cose_message_create() and t_cose_message_decode handle
88 * COSE_Message. This is for handling COSE messages that might be signed,
89 * encrypted, MACed or some combination of these. In the simplest case
90 * they decode the CBOR tag number and switch off to one of the
91 * above handlers. In more complicated cases they recursively handle
92 * nested signing, encrypting and MACing. (Lots of work to do on
93 * this…)
94 *
95 *
96 * Headers
97 *
98 * t_cose_headers_decode() and t_cose_headers_encode() handle the
99 * protected and unprotected header parameter buckets that are used by
100 * all the COSE messages.
101 *
102 * This also defines a data structure, t_cose_header_parameter that
103 * holds one single parameter, for example an algorithm ID or a
104 * kid. This structure is used to pass the parameters in and out of
105 * all the methods above. It facilitates the general header
106 * parameter implementation and allows for custom and specialized
107 * headers.
108 *
109 *
110 * COSE_signature
111 *
112 * t_cose_signature_sign and t_cose_signature_verify are abstract
113 * bases classes for a set of implementations of COSE_Signature. This
114 * design is chosen because there are a variety of signature schemes
115 * to implement. Mostly these correspond to different signing
116 * algorithms, but there is enough variation from algorithm to
117 * algorithm that the use of an abstract base class here makes sense.
118 *
119 * Currently there is a "main" signer/verify that supports RSA
120 * and ECDSA. There is another one for EdDSA, because it is
121 * structurally different in not using a hash. Future
122 * signer/verifiers might include one for counter signatures
123 * and one for PQ.
124 *
125 * The user of t_cose will create instances of t_cose_signature and
126 * configure them into t_cose_sign_sign() and t_cose_sign_verify().
127 *
128 *
129 * COSE_recipient
130 *
131 * t_cose_recipient_enc and t_cose_recipient_dec are abstract base
132 * classes for the set of concrete implementations of
133 * COSE_recipient. Because the variation in one type of COSE_recipient
134 * to another is so varied, this is whe *re the abstract base class is
135 * necessary.
136 *
137 *
138 * Notes on objects
139 *
140 * Note that this use object-orientation here gives some very nice
141 * modularity and extensibility. New types of COSE_recipient can be
142 * added to COSE_Encrypt and COSE_Mac without changing their
143 * implementation at all. It is als *o possible to add new types of
144 * recipients without even modifying the main t_cose library.
145 *
146 * This effectively gives dynamic linking for a lot of code that
147 * makes dead-stripping by the linker more effective and requires
148 * less use of #defines to reduce object code size. For example,
149 * if a switch were used to select EdDSA, all the EdDSA code would
150 * always be linked unless it was #ifdef'd out. With this design
151 * not calling the EdDSA signer init function removes all reference
152 * to EdDSA and it will be dead-stripped.
153 *
154 * This design should faciliate a lot of variance and innovation
155 * in signers and encryptors, for example faciliating key
156 * database look ups, use of certificates, counter signatures
157 * and such, all without changing the source or even object
158 * code of the core t_cose library.
159 *
160 * COSE_Key
161 *
162 * Some formats of COSE_recipient have parameters that are in the
163 * COSE_key format. It would be useful to have some library code to
164 * handle these, in particular to encode and decode from the key data
165 * structure used by the crypto library (OpenSSL, PSA, …).
166 */

842/* Structure that holds all the inputs for signing that is
843 * used in a few places (so it ends up in t_cose_common.h).
844 * It is public because it is part of the signer/verify
845 * call back interface. It is also used for MAC.
846 *
847 * These are the inputs to create a Sig_structure
848 * from section 4.4 in RFC 9052.
849 *
850 * ext_sup_data and sign_protected may be \ref NULL_Q_USEFUL_BUF_C.
851 *
852 * payload is a CBOR encoded byte string that may
853 * contain CBOR or other.
854 *
855 * body_protected are the byte-string wrapped protected
856 * header parameters from the COSE_Sign or COSE_Sign1.
857 */
858struct t_cose_sign_inputs {
859 struct q_useful_buf_c body_protected;

886/* This is the base class for all implementations
887 * of COSE_Signature and COSE_Recipient. It implments what
888 * is common to them:
889 * - The ability to identify the type of one

892 * The linked list part saves object code because the same
893 * function to add to the linked list is used for all types

897 * instance to be able to call some special methods it
898 * might implement, particularly for COSE_Signature verifiers

900 * and COSE_Encrypt decryptor loops over a set of them.
